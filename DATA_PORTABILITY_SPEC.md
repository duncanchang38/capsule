## 1. Format Versioning Strategy

### Decision: Monotonic integer, `metadata.json` is authoritative

The export format uses a single monotonically-increasing **integer** as its schema version. Semver is inappropriate — consumers only need to know "can I read this file?", a single dimension. Date-based versions (`20260406`) are cumbersome to compare in code and ambiguous across timezones. Integer versions (`1`, `2`, `3`) are unambiguous and trivially comparable.

### Version placement

The version appears in exactly two locations:

1. **`metadata.json`** → `"schema_version": 1` — the canonical source of truth.
2. **The ZIP filename** → `capsule-export-v1-20260406T143000Z.zip` — advisory only, for human inspection before unzipping.

**`metadata.json` always wins.** If the filename says `v1` and `metadata.json` says `2`, trust `metadata.json`. The filename can be renamed; `metadata.json` cannot be modified without breaking the checksum.

### Compatibility model

| Condition | Schema version change |
|---|---|
| Add optional field to `CaptureMetadata` | None — backward compatible |
| Add new `capture_type` enum value | None — see enum policy below |
| Remove a field or change a field's type | **Increment version** |
| Rename a field | **Increment version** |
| Restructure ZIP layout | **Increment version** |

### Unknown enum value policy

Importers **must** tolerate unknown `capture_type`, `completion_type`, and `status` values — do not reject or warn. Unknown `capture_type` values are preserved verbatim and treated as `inbox` for UI routing purposes only. This is not silent data loss; the original value is preserved in the record.

### Forward compatibility (consumers)

Importers must treat unknown fields as ignorable. A v1 importer can safely process a v1 export generated by a newer build with additional optional fields.

A v1 importer receiving a v2 export **must** reject with a clear error:
```
"This export requires importer version ≥ 2. Please update Capsule."
```

### Backward compatibility (exporters)

When version 2 ships, the exporter supports `?format_version=1` for users sharing exports with older Capsule instances. This is supported for **one major version back only**. If a field in v2 cannot be represented in v1 (e.g., a renamed field), the exporter drops it and adds a `"downgrade_warnings"` array to `metadata.json` listing the omitted data.

### Version negotiation at import time

```
1. Read metadata.json → schema_version
2. If schema_version == IMPORTER_MAX_VERSION → proceed
3. If schema_version < IMPORTER_MAX_VERSION → apply migration chain (§8)
4. If schema_version > IMPORTER_MAX_VERSION → reject 422 + upgrade message
```

## 2. ZIP Archive Structure

```
capsule-export-v1-20260406T143000Z.zip
└── capsule-export/
    ├── metadata.json              # Last file written — root of trust
    ├── schema.json                # Bundled schema (documentation only)
    ├── captures.jsonl             # All captures, one JSON object per line
    ├── entities.jsonl             # Extracted entities (GraphRAG layer)
    └── topics/
        ├── habits.json
        ├── machine-learning.json
        └── _uncategorized.json    # Captures with no topic
```

### Writing order

`metadata.json` **must be the last file written** to the ZIP. An importer that finds a ZIP where `metadata.json` is absent or unparseable must reject the entire archive. This ensures a crash-interrupted export is always detectable.

### Rationale for flat `captures.jsonl`

All captures in one JSONL file enables O(1)-memory streaming for any size export. Topic files in `topics/` are pre-computed summaries for human inspection; they are not the source of truth for topic membership and are not used by the importer.

### File naming rules

Topic filenames: canonical topic name → lowercase → spaces to hyphens → strip non-alphanumeric non-hyphen characters. `"Machine Learning"` → `machine-learning.json`. Slug collisions (e.g. `"AI/ML"` and `"AI-ML"` both → `ai-ml`) are resolved by appending `_2`, `_3`, etc., with a `"slug_collision": true` flag inside the file.

### Bundled `schema.json`

The `schema.json` bundled in the ZIP is **documentation only**. Importers always use their own built-in schema keyed on `schema_version` from `metadata.json`. The bundled schema is present so a human can inspect an old export and understand its structure without access to the Capsule codebase.

## 3. `metadata.json` Specification

This file is always read first by any importer. It must be present and valid or the import is rejected. It is written **last** during export generation.

### Design principle: zero PII

The export contains **no user identity information**. It is a portable document of your captures — nothing more. It can be shared, inspected, backed up, or imported into any Capsule account without leaking who it belongs to.

### Field definitions

| Field | Type | Required | Description |
|---|---|---|---|
| `schema_version` | `integer` | Yes | Export format version. Currently `1`. |
| `app_version` | `string` | Yes | Capsule application semver that generated this export, e.g. `"2.4.1"`. |
| `exported_at` | `string` | Yes | ISO 8601 UTC, e.g. `"2026-04-06T14:30:00Z"`. All datetimes in this spec are UTC with `Z` suffix. |
| `total_captures` | `integer` | Yes | Total lines in `captures.jsonl`. Importers validate this before processing. |
| `total_topics` | `integer` | Yes | Count of topic files in `topics/`. |
| `total_entities` | `integer` | Yes | Total lines in `entities.jsonl`. `0` if entity extraction has not run. |
| `capture_types_present` | `string[]` | Yes | Deduplicated list of `capture_type` values. |
| `includes_deleted` | `boolean` | Yes | Whether captures with `status = "deleted"` are included. |
| `includes_archived` | `boolean` | Yes | Whether captures with terminal statuses are included. |
| `generated_by` | `string` | Yes | Always `"capsule"`. Reserved for third-party tools. |
| `checksums` | `object` | Yes | SHA-256 hex digest keyed by filename. See §6. |
| `downgrade_warnings` | `string[]` | No | Present only when `?format_version` downgrade was used. Lists omitted or truncated fields. |

### Example

```json
{
  "schema_version": 1,
  "app_version": "2.4.1",
  "exported_at": "2026-04-06T14:30:00Z",
  "total_captures": 847,
  "total_topics": 23,
  "total_entities": 1203,
  "capture_types_present": ["to_hit", "to_learn", "to_cook", "to_know", "calendar"],
  "includes_deleted": false,
  "includes_archived": true,
  "generated_by": "capsule",
  "checksums": {
    "captures.jsonl": "a3f2c1d9e4b8f7a6c5d2e1f0b9a8c7d6e5f4a3b2c1d0e9f8a7b6c5d4e3f2a1b0",
    "entities.jsonl": "b2e1d0c9f8a7b6c5d4e3f2a1b0c9d8e7f6a5b4c3d2e1f0a9b8c7d6e5f4a3b2"
  }
}
```

### Metadata sub-key classification

Fields prefixed with `_` in `CaptureMetadata` are internal application state or live database pointers. Importers must handle them as follows:

| Sub-key | Type | On import |
|---|---|---|
| `source_id` | DB pointer (sprint parent) | Remap to new DB id; clear if not in batch |
| `sprint_ids` | DB pointer array (sprint children) | Remap each to new DB id; clear unreachable |
| `sprint_count`, `sprint_index`, `sprint_total` | Derived counts | Preserve as-is |
| `source_title` | Denormalised string | Preserve as-is |
| `merge_suggestion` | DB pointer | Clear — cannot be reliably remapped |
| `deleted_at`, `archived_at` | Internal state timestamps | Preserve as-is |
| `deferred_count` | Internal counter | Preserve as-is |

All other fields (`topic`, `url`, `resource_type`, `author`, etc.) are safe to copy verbatim.

## 5. `captures.jsonl` Format

### Format: JSON Lines

One complete JSON object per line, UTF-8, `\n` line endings. Never `\r\n`.

Advantages over a JSON array: O(1)-memory streaming reads; corruption isolates to a single line; append-friendly for incremental exports.

### `export_id` — intra-archive reference only

Every capture has an `export_id`: a random **UUIDv4** generated fresh at export time.

```python
import uuid

def make_export_id() -> str:
    return str(uuid.uuid4())
```

Its only job is to link records **within the ZIP** — topic files and entity files reference captures by `export_id`. It is discarded on import; the database assigns new integer primary keys.

This means:
- No `export_id` column on the `captures` table
- No server-side seed or `settings` table
- Importing the same ZIP twice creates duplicates — acceptable for a personal app. The UI can detect this by checking `exported_at` in `metadata.json` and warning the user.

The original DB PK is preserved as `metadata._source_id` for debugging only. Import logic never uses it.

### Field definitions

| Field | Type | Required | Notes |
|---|---|---|---|
| `export_id` | `string` (UUID) | Yes | Stable globally-unique identifier. Primary key for import. |
| `capture_type` | `string` (enum) | Yes | Unknown values preserved verbatim, routed as `inbox`. |
| `completion_type` | `string` (enum) | Yes | |
| `content` | `string` | Yes | Never empty. |
| `summary` | `string` | Yes | AI-generated. |
| `status` | `string` (enum) | Yes | |
| `deadline` | `string \| null` | No | `YYYY-MM-DD` date. No timezone — it is a calendar date, not a point in time. |
| `notes` | `string \| null` | No | May contain HTML. Importers sanitize on ingest. |
| `metadata` | `object` | Yes | Always present; may be `{}`. |
| `created_at` | `string` | Yes | ISO 8601 UTC. Must end with `Z`. e.g. `"2026-03-15T09:22:00Z"` |
| `updated_at` | `string` | Yes | ISO 8601 UTC. Must end with `Z`. |

## 6. Integrity — Checksums

`metadata.json` contains a `checksums` object with SHA-256 hex digests for every named file in the archive (excluding `metadata.json` itself, since it is the root of trust).

```json
"checksums": {
  "captures.jsonl":  "a3f2c1d9...",
  "entities.jsonl":  "b2e1d0c9...",
  "schema.json":     "c1d0e9f8..."
}
```

Topic files in `topics/` are **not** individually checksummed — they are derived summaries. Their integrity is guaranteed if `captures.jsonl` is intact.

**Importer validation order:**
1. Unzip to temp directory
2. Parse `metadata.json` — reject if absent or unparseable
3. Verify SHA-256 of each file listed in `checksums` — reject on mismatch
4. Verify line count of `captures.jsonl` matches `total_captures` — reject on mismatch
5. Proceed with import

## 7. `entities.jsonl` Format

Entity records link extracted knowledge graph nodes back to captures. One JSON object per line.

| Field | Type | Required | Notes |
|---|---|---|---|
| `capture_export_id` | `string` (UUID) | Yes | References `Capture.export_id`. |
| `entity_type` | `string` | Yes | e.g. `person`, `place`, `concept`, `technology`, `organization` |
| `value` | `string` | Yes | The extracted entity text. |
| `confidence` | `number \| null` | No | 0.0–1.0 |

On import: insert entities and rebuild the entity graph. For **foreign imports**, trigger background re-extraction instead of copying entities verbatim — entity quality varies by model version and extracted entities may reference context specific to the source instance.

If `total_entities` is `0`, `entities.jsonl` may be an empty file or absent. Both are valid.

## 8. `topic.json` Specification

One file per topic in `topics/`. Summaries only — importers reconstruct topic membership from `metadata.topic` on each capture.

| Field | Type | Required | Description |
|---|---|---|---|
| `name` | `string` | Yes | Filename-safe slug. |
| `canonical_name` | `string` | Yes | Display name. |
| `capture_count` | `integer` | Yes | |
| `capture_export_ids` | `string[]` (UUID) | Yes | Most recent first. References `Capture.export_id`. |
| `first_captured_at` | `string \| null` | No | UTC with `Z`. |
| `last_captured_at` | `string \| null` | No | UTC with `Z`. |
| `slug_collision` | `boolean` | No | `true` if slug was suffixed to avoid collision. |

### `_uncategorized.json`

Captures with `metadata.topic == null` are collected here.

## 9. Import/Export API

All endpoints require authentication (`Authorization: Bearer <token>` or session cookie).

### Export

```
GET /api/export
```

The same endpoint covers both full export and single-capture export (inspired by Dify's single-resource DSL export pattern: `GET /apps/<id>/export`). The file format is identical in both cases — a ZIP with `metadata.json`, `captures.jsonl`, and `entities.jsonl`. A single-capture export is just a ZIP with one line in `captures.jsonl`.

| Parameter | Type | Default | Description |
|---|---|---|---|
| `format_version` | `integer` | `1` | Target schema version for downgrade compatibility. |
| `capture_id` | `integer` | — | Export a single capture by DB id. Mutually exclusive with `topic`/`since`. |
| `include_deleted` | `boolean` | `false` | Include `status = "deleted"` captures. Ignored when `capture_id` is set. |
| `include_archived` | `boolean` | `true` | Include terminal-status captures. |
| `topic` | `string` | — | Restrict to one topic (slug). Optional. |
| `since` | `string` | — | ISO date. Export captures created on or after this date. Optional. |

**`query` captures are never exported.** The `query` capture type is internal (search sessions, not user content). It is excluded from all exports with an implicit `WHERE capture_type != 'query'` clause. This is noted in the appendix.

**Response:** `200 OK`, `Content-Type: application/zip`.

**Error responses:**

| Code | Condition |
|---|---|
| `400` | `format_version` not supported |
| `401` | No valid session |
| `429` | Rate limit: 3 exports/user/hour. Header: `Retry-After` |

> **Note:** Async export jobs (202 + polling) are a **future milestone** once a job queue exists. For v1, all exports are synchronous. If this becomes a timeout risk, the threshold for async can be added then.

### Import

```
POST /api/import
Content-Type: multipart/form-data
```

| Parameter | Type | Default | Description |
|---|---|---|---|
| `dry_run` | `boolean` | `false` | Validate and preview without writing. |

**IDs are always regenerated on import.** Every capture receives a new database primary key. No conflict modes needed. `_sprint_ids` and `_merge_suggestion` internal pointers are remapped to the new IDs where the referenced captures are present in the same import batch; otherwise they are cleared.

**Import processing order:**
1. Unzip to temp directory
2. Parse and validate `metadata.json`
3. Validate `schema_version` is within supported range
4. Verify checksums for all files in `checksums`
5. Verify `captures.jsonl` line count matches `total_captures`
6. Stream-parse `captures.jsonl` line by line, validate, collect errors (max 50)
7. If `dry_run=true` → return summary and stop
8. Build an `export_id → new_db_id` map for the entire batch (needed to remap internal pointers)
9. Begin DB transaction. Insert captures in batches of 500 with fresh DB IDs. Remap `_sprint_ids` and `_merge_suggestion` using the map; clear any that cannot be resolved
10. Reconstruct topics from `metadata.topic` on each capture
11. Insert entities from `entities.jsonl`, remapping `capture_export_id` → new DB id
12. Commit. Return summary.

**Successful response:**
```json
{
  "imported": 843,
  "skipped": 4,
  "failed": 0,
  "topics_created": 18,
  "topics_merged": 5,
  "entities_imported": 1203,
  "entities_queued_for_reextraction": 0,
  "warnings": [
    "4 captures with unknown capture_type 'to_watch' were imported with type preserved.",
    "Capture 550e8400-... skipped (conflict mode: skip).",
    "_sprint_ids remapped for 12 captures. 3 sprint references could not be resolved and were cleared."
  ],
  "dry_run": false
}
```

**Error responses:**

| Code | Condition | Body |
|---|---|---|
| `400` | Not a ZIP, missing `metadata.json`, unparseable `captures.jsonl` | `{"error": "invalid_archive", "detail": "..."}` |
| `409` | `overwrite` would affect >100 captures without `confirm_overwrite=true` | `{"error": "destructive_import", "overwrite_count": 103}` |
| `413` | ZIP exceeds 50MB | `{"error": "file_too_large", "max_bytes": 52428800}` |
| `422` | Schema version not supported | `{"error": "unsupported_schema_version", "found": 3, "max_supported": 1}` |
| `422` | Checksum mismatch | `{"error": "checksum_mismatch", "file": "captures.jsonl", "expected": "...", "actual": "..."}` |
| `401` | No valid session | — |

## 10. Migration Pattern

### Non-breaking change (no version bump)

Adding an optional field to `CaptureMetadata` — e.g., `og_image`. Since the metadata bag is open (`additionalProperties: true`) and the field is optional, existing importers ignore it. No version bump needed.

**Checklist for non-breaking:**
- [ ] Field is optional (nullable or has a default)
- [ ] Field does not alter the semantics of existing fields
- [ ] Existing importers produce valid (if incomplete) data without it

### Breaking change (version bump required)

**Example: v1 → v2 — rename `content` → `raw_input` and `summary` → `title`**

```python
def migrate_v1_to_v2(capture: dict) -> dict:
    capture = dict(capture)
    if "content" in capture and "raw_input" not in capture:
        capture["raw_input"] = capture.pop("content")
    if "summary" in capture and "title" not in capture:
        capture["title"] = capture.pop("summary")
    return capture

def migrate_v2_to_v1(capture: dict) -> dict:
    capture = dict(capture)
    if "raw_input" in capture and "content" not in capture:
        capture["content"] = capture.pop("raw_input")
    if "title" in capture and "summary" not in capture:
        capture["summary"] = capture.pop("title")
    return capture
```

### Migration chain

```python
MIGRATIONS = {
    (1, 2): migrate_v1_to_v2,
    (2, 3): migrate_v2_to_v3,
}

def migrate_capture(capture: dict, from_version: int, to_version: int) -> dict:
    for v in range(from_version, to_version):
        fn = MIGRATIONS.get((v, v + 1))
        if fn:
            capture = fn(capture)
    return capture
```

A v1 export imported into a v3 importer applies `v1→v2` then `v2→v3` in sequence. No direct v1→v3 shim is ever needed.

**User-facing migration notice:**
```json
{
  "imported": 843,
  "warnings": [
    "Export was schema version 1. Automatically migrated to version 2. Fields renamed: content → raw_input, summary → title."
  ]
}
```

## 11. Edge Cases

### Captures with no URL

`metadata.url` is `null` for non-URL captures. Valid and expected. Importers must never reject or warn on null URLs.

Downstream consumers needing a simplified type taxonomy can derive it:

```
to_hit | calendar           → todo
to_learn + url != null      → url bookmark
to_learn + url == null      → note
to_cook | to_know           → note
```

### Deleted topics

Topics are derived from `metadata.topic` on captures — there is no topics table. A "deleted topic" means all its captures were deleted or reassigned. If `include_deleted=false` (default), deleted captures are excluded; if a topic then has zero captures, its topic file is not written. The import reconstructs only the topics that have surviving captures.


### Large exports

- JSONL streaming means export memory usage is bounded: the exporter writes to disk as it reads from the DB (500-row cursor batches)
- Import streams `captures.jsonl` line by line, inserts in 500-record transaction batches
- At typical capture sizes (2–5 KB each uncompressed), 10,000 captures ≈ 5–10 MB compressed — well within the 50MB limit
- If the import exceeds 30 seconds, queue it as a background job (future milestone)

### Best-effort import on errors

Rather than aborting on the first invalid line, the importer collects up to 50 validation errors, imports valid records, and reports failures. This prevents a single corrupt line from discarding thousands of valid records on large imports.

### `notes` HTML sanitization

The `notes` field may contain HTML from the rich text editor. Importers **must** sanitize on ingest using an allowlist approach:

- **Allow:** `p`, `br`, `strong`, `em`, `ul`, `ol`, `li`, `h1`–`h4`, `blockquote`, `code`, `pre`, `a` (href with `http`/`https` only)
- **Strip:** `script`, `style`, `iframe`, `object`, `embed`, `form`, `input`, all `on*` event attributes, `javascript:` hrefs, `data:` URIs

### Timezone handling

All datetime fields (`created_at`, `updated_at`, `exported_at`, etc.) **must** be serialized as UTC with a `Z` suffix: `"2026-04-06T14:30:00Z"`. The `deadline` field is a calendar date only (`YYYY-MM-DD`) with no timezone component. Importers that receive naive ISO strings (no timezone indicator) should assume UTC.

## Appendix: What Is Not Exported

| Data | Reason |
|---|---|
| `capture_type = 'query'` captures | Internal search sessions, not user content |
